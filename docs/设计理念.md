# 设计理念

社区服务平台的设计目标是：让居民“想办事就能立刻办”，让管理员“看得清、管得快、可追溯”。为此，我们在交互、信息架构与技术实现上采用了“以流程为中心”的整体设计思路。

## 1. 以居民为中心：一站式入口与低学习成本
- **服务卡片驱动**：用户在主页以“场地查询/预约/报修/通知/便民服务”等卡片入口完成决策，减少菜单层级。
- **页面内切换**：采用动态内容切换而不是频繁跳转，降低打断感，并让用户形成稳定的操作路径记忆。
- **返回与状态提示**：每个功能模块提供返回主页能力，并通过 Toast 提示关键操作结果（加载/提交/失败/成功）。

## 2. 可用性优先：把“状态”做成一眼能懂
社区服务的关键在于“某个资源是否可用、是否冲突、当前处于什么阶段”。因此平台在呈现层注重：
- 场地状态可视化：可用/占用中/维护中等信息直接展示
- 预约冲突可控：无论通过表单还是 AI 辅助，都会在创建时进行冲突校验并反馈原因
- 工单/通知状态表达：报修、通知、预约等关键对象以状态标签形式呈现

## 3. 自然交互：AI 助手作为“办事导航”
本项目的 AI 助手并非仅回答闲聊，而是围绕社区事务的典型动作进行“意图识别 + 任务执行”：
- 识别预约意图：自动提取场地/日期/时间，并在冲突时引导调整
- 识别查询意图：按场地与日期返回可用性/预约情况
- 识别取消意图：列出可取消对象并执行状态变更

这样设计的出发点是：用户不需要记住系统术语或操作路径，只要描述诉求即可获得“下一步指令”。

## 4. 信息架构：双端分工清晰，权限模型可扩展
- **用户端**：关注“我能做什么/当前状态/我的记录”
- **管理员端**：关注“审核、变更、运营统计、系统维护”

当前实现未加入复杂鉴权（便于展示与演示），但模块边界清晰，后续可在“路由层/前端路由层”逐步补齐权限控制。

## 5. 工程取舍：演示友好优先，预留可演进空间
为了便于展示与本地运行，本项目在工程实现上采取了较轻量的策略：
- 数据层使用文件化持久化（可读、可演示）
- ASR 部分为示例/模拟（语音输入可来自浏览器 Web Speech API，或来自 ESP32 设备端的识别文本/控制事件，取决于接入的语音/协议网关实现）
- 统计/趋势接口包含模拟数据（展示仪表盘效果）

这些取舍保证了“短周期交付价值”，同时为后续演进（替换为真实 SQLite/接入真实 ASR/完善统计口径）保留升级空间。

## 6. 设计原则总结
- 面向任务：围绕预约、报修、查询等动作组织界面与接口
- 可视化反馈：让用户知道“请求发生了什么、结果如何”
- 低门槛交互：卡片入口 + AI 导航 + 语音输入（可来自浏览器或 ESP32 设备端，取决于接入方案）
- 可扩展架构：模块化路由与服务层，便于后续替换与增强

## 7. 设备端扩展的标准化
当你把 ESP32 硬件作为“语音入口 + 控制端”接入时，建议遵循三层分工：
- 协议层标准：底层通信选择 WebSocket 或 MQTT + UDP，并在设备侧与网关侧约定消息时序
- 控制层标准：使用 MCP 的 `initialize / tools/list / tools/call` 统一“工具发现与调用”
- 业务层适配：将 MCP 工具映射到本仓库现有 HTTP API（预约/报修/设备控制），再回填执行结果
