# 作品应用场景

社区服务平台面向“社区居民日常事务高频、分散、信息不对称”的痛点，提供一站式入口：场地预约、AI 助手咨询、设备报修、通知公告与便民服务查询，并配套管理员端完成审核与运维闭环。

## 1. 场地与设施自助预约
- 目标用户：居民/社团负责人
- 主要痛点：想用体育/会议/学习空间，但不了解实时状态与可用时段
- 解决方式：
  - 用户可通过“场地查询”查看可用/维护/占用等状态
  - 通过“场地预约”表单选择时间并提交预约（或使用 AI 让助手完成预约指引）
  - AI 助手会根据输入识别预约意图，自动匹配场地并进行时间冲突检查
- 价值体现：减少人工沟通与重复排队，提高场地使用效率

## 2. AI 对话助手实现“自然语言办事”
- 目标用户：不熟悉系统操作的居民，或希望快速解决问题的人群
- 主要痛点：传统表单/菜单路径多，用户不知道该点哪里、怎么描述需求
- 解决方式：
  - 用户直接用自然语言提出需求（预约/查询/取消）
  - 助手基于社区服务语境给出可执行建议，并在需要时触发预约/取消流程
  - 对于模糊输入（如“会议室”）可自动补全默认场景
- 价值体现：降低使用门槛，增强“随问随答”的交互体验

## 3. 设备报修与运维工单闭环
- 目标用户：所有需要报修的居民
- 主要痛点：报修信息不标准、缺少场地/优先级，导致处理效率低
- 解决方式：
  - 用户可在“设备报修”中提交工单（标题、描述、场地、优先级）
  - 管理员在报修管理中查看、处理状态变更
  - 平台把报修信息结构化存储，便于追踪与统计
- 价值体现：统一报修信息，提高响应效率与可追溯性

## 4. 政策通知与便民服务信息触达
- 目标用户：社区所有居民
- 主要痛点：社区通知发布后难以被及时发现；信息来源分散
- 解决方式：
  - 用户端展示已发布通知，可按类型筛选（公告/服务/活动/安全等）
  - 便民服务从服务列表加载并展示关键字段（提供商、描述、评分、状态）
- 价值体现：降低信息搜寻成本，让居民更容易找到“需要的服务”

## 5. 管理员端的综合运营与审核管理
- 目标用户：社区管理员/运维人员
- 主要痛点：需要同时处理用户、预约、场地与报修的多类数据
- 解决方式：
  - 管理端提供侧边栏导航与内容区动态切换
  - 支持场地管理（状态变更）、预约管理（审核/状态更新）、用户管理
  - 支持通知与服务管理，维护社区服务生态的有效性
- 价值体现：把分散管理工作集中化，提高管理效率

## 6. 设备监控与紧急控制（示例）
- 目标用户：管理员/值班人员
- 主要痛点：需要快速了解设备状态，并在紧急情况下执行统一动作
- 解决方式（当前为示例接口形态）：
  - 设备状态分类查询（房间/系统）
  - 控制类接口（例如空调系统/照明系统/批量控制/紧急关闭与重置）
- 价值体现：让系统具备扩展到“物联网运维”场景的接口形态

## 7. ESP32 语音入口（扩展）
- 目标用户：居民、值班人员
- 主要痛点：部分用户不熟悉网页操作路径，希望通过“说出需求”完成预约/报修/查询；运维人员希望用语音快速执行状态切换或紧急操作。
- 实现思路（接入前提）：
  - ESP32 设备端通过 WebSocket 或 MQTT + UDP 与“协议网关/服务器”建立会话
  - 在会话中利用 MCP 工具调用完成“意图 -> 工具 -> 业务接口”的闭环
  - 网关侧把 MCP `tools/call` 映射到本仓库 HTTP API（如 `/api/reservation/create`、`/api/repair/create`、`/api/device/...`），并回填执行结果给设备侧继续后续流程
- 价值体现：
  - 降低学习成本：用自然语言完成社区事务
  - 提升响应效率：紧急场景下可快速触发设备状态变更
  - 增强可扩展性：后续可新增更多 MCP 工具来扩展“语音控制范围”
