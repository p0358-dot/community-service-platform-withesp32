# 技术方案

本节从架构视角说明社区服务平台的实现方式：前端如何与后端通信、后端如何组织路由与业务逻辑、AI 助手如何触发预约/查询/取消、以及数据如何持久化。

## 1. 总体架构

系统采用“前后端分离（前端为静态页面）+ Express API”的结构：
- **前端**：原生 HTML/CSS/JavaScript（用户端 `frontend/user/*`，管理员端 `frontend/admin/*`）
- **后端**：Node.js + Express
- **数据层**：`backend/database/db.js` 文件化持久化（JSON 文件 `community.db`），并非使用真正的 SQL 数据库
- **AI 服务**：`backend/services/aiService.js` 调用阿里云 DashScope（兼容备用 OpenAI 配置）
- **设备端（ESP32，可选）**：设备侧可通过 `websocket` 或 `mqtt + udp` 与“协议网关/服务器”通信，并通过 `type: "mcp"` 进行工具调用（协议细节见 `esp32/docs/*`）。

请求流（通用）：
1. 浏览器发起 HTTP 请求（`fetch`）
2. Express 路由匹配（`/api/...`）
3. 路由调用服务层（如 AI）或数据库层（db.js）
4. 返回 JSON 响应给前端，前端渲染 UI

## 2. 后端模块划分

### 2.1 服务器入口：`backend/server.js`
职责包括：
- 配置 CORS 与 body 解析
- 提供静态文件：`express.static(.../frontend)`
- 挂载 API 路由：
  - `/api/ai` -> `backend/routes/ai.js`
  - `/api/user` -> `backend/routes/user.js`
  - `/api/reservation` -> `backend/routes/reservation.js`
  - `/api/venue` -> `backend/routes/venue.js`
  - `/api/device` -> `backend/routes/device.js`
  - `/api/statistics` -> `backend/routes/statistics.js`
  - `/api/notice` -> `backend/routes/notice.js`
  - `/api/repair` -> `backend/routes/repair.js`
  - `/api/service` -> `backend/routes/service.js`
- 提供健康检查：`GET /api/health`
- 兜底错误处理中间件

### 2.2 业务路由层：`backend/routes/*`
每个资源域（场地、预约、报修、通知、服务、设备、统计、用户、AI）由独立路由文件承载：
- `routes/venue.js`：场地列表、详情、创建/更新/删除、状态更新
- `routes/reservation.js`：预约创建、取消、列表、可用时间计算（按时间段是否被预约）
- `routes/repair.js`：报修列表、用户报修列表、报修详情、创建、状态更新
- `routes/notice.js`：通知列表、已发布通知、详情、创建、状态更新
- `routes/service.js`：服务列表、活跃服务、详情、创建、状态更新
- `routes/device.js`：房间/系统设备状态查询、设备控制、批量控制、统计、维护记录、紧急操作（示例）
- `routes/statistics.js`：概览与多维统计/趋势数据（部分为模拟数据用于展示）
- `routes/user.js`：用户列表/详情/创建/更新/删除、用户统计概览
- `routes/ai.js`：AI 对话、状态、历史、会话列表、取消预约、测试连接等

### 2.3 AI 业务服务：`backend/services/aiService.js`
核心流程：
1. `chat(message, history, userId, sessionId)` 入口
2. `analyzeIntent(message)`：使用关键词启发式方法识别 intent（booking/query/cancel/chat）
3. 根据 intent 路由到：
   - `handleBookingIntent`：预约意图 -> 场地匹配 -> 获取该场地当日预约 -> 冲突检测 -> 创建预约
   - `handleQueryIntent`：查询意图 -> 场地匹配 -> 查询预约记录 -> 输出可用性/预约情况
   - `handleCancelIntent`：取消意图 -> 拉取用户预约 -> 返回待取消列表/执行取消
4. 对于普通聊天：`buildMessages` 组织 system prompt + history，再通过 `callDashScope` 发起请求

AI 请求模型选择与鉴权：
- 通过 `backend/config.js` 读取 `DASHSCOPE_API_KEY`、`DASHSCOPE_MODEL`、`DASHSCOPE_API_URL`
- 使用 axios 调用 DashScope 兼容接口：`.../chat/completions`

### 2.4 数据层：`backend/database/db.js`
实现特征：
- 内部维护 `this.data`，包含 users、venues、reservations、devices、repairs、notices、services、chatMessages 等集合
- 初始化：
  - 如果 `community.db` 存在，读取并解析 JSON，重建集合与 nextId
  - 不存在则写入初始示例数据（场地/设备/用户/预约/通知/服务/聊天记录）
- 持久化：
  - 每次数据变更后调用 `saveData()` 将 JSON 写回 `community.db`
- 聊天会话：
  - `saveChatMessage` 追加消息
  - `getChatHistory` 按 userId + sessionId 返回历史
  - `getChatSessions` 统计会话最近活跃时间并排序

### 2.5 语音设备端（ESP32）通信与 MCP 能力（扩展）

`esp32/` 硬件项目通常作为“语音入口 + 物联网控制端”，其设备侧协议与工具控制能力可以概括为：

- 底层通信：
  - WebSocket：在握手阶段发送 `type: "hello"`（`transport: "websocket"`），建立会话后发送二进制 Opus 音频帧与 JSON 控制消息。
  - MQTT + UDP：MQTT 通道用于控制/状态 JSON，UDP 通道用于加密 Opus 音频数据传输。
- 物联网控制（MCP）：
  - 设备侧通过 `type: "mcp"` 承载 JSON-RPC 2.0 的 `initialize / tools/list / tools/call`。
  - 后台/网关侧需要实现 MCP 客户端，发现设备工具并按工具名与参数调用，拿到结果再继续后续流程。

与本仓库的能力边界：当前社区服务后端主要提供 HTTP API 与 DashScope AI 对话接口，并不直接实现 WebSocket/MQTT/MCP 的会话与工具回调。
如果你需要把设备的 MCP 工具调用落到社区业务（预约/报修/设备控制），一般做法是在后端旁新增“协议适配网关”，将 MCP 的 `tools/call` 映射到本仓库现有 HTTP API。

## 3. 关键用例的数据流

### 3.1 AI 预约（自然语言办事）
1. 前端在 AI 对话框提交文本
2. 请求 `POST /api/ai/chat`
3. `routes/ai.js`：保存用户消息 -> 调用 `aiService.chat(...)`
4. `aiService.chat(...)`：
   - `analyzeIntent` -> booking
   - `handleBookingIntent`：
     - `db.getAllVenues` 匹配场地
     - `db.getReservationsByVenue(venueId, date)` 获取该日期已预约时段
     - 冲突判断（同 slot 已存在 -> 返回冲突提示）
     - 无冲突 -> `db.createReservation`
5. 返回给前端消息文本 + 预约编号等信息

### 3.2 用户取消预约
路径有两种：
- 通过 AI 对话（`POST /api/ai/cancel`）
- 通过我的预约页面手动取消（前端调用 `DELETE /api/reservation/:id`）

数据变更点：`db.updateReservationStatus(..., 'cancelled', ...)`

### 3.3 报修工单闭环
1. 用户提交报修（`POST /api/repair/create`）
2. 管理员在报修管理页面拉取列表（`GET /api/repair/list`）
3. 管理员通过状态下拉更新工单（`PUT /api/repair/:id/status`）

## 4. API 与响应约定

不同路由文件在实现上使用了相似的约定：
- 成功返回：`{ success: true, data: ... }` 或直接 `res.json({ success: true, data, message })`
- 错误返回：HTTP 500/400 + `success: false` 或 `error/message` 字段

建议在后续迭代中统一响应结构，方便前端减少“兼容不同响应格式”的代码。

## 5. 部署与运行

部署方式（本地示例）：
1. 配置 `.env`
2. `npm install`
3. `npm start`
4. 访问：
   - `http://localhost:3000/user/`
   - `http://localhost:3000/admin/`

后端服务监听端口由 `config.js` 控制（默认 `3000`）。
服务器在启动时会显示健康检查与 AI 接口地址。

## 6. 安全性与可靠性（当前实现与建议）

当前风险点（用于方案说明，可在答辩中作为“改进方向”）：
- **无身份认证**：所有接口公开访问，生产环境应接入登录/鉴权、区分用户/管理员权限
- **CORS 配置宽松**：开发方便，但需根据部署域名收敛策略
- **密钥安全**：后端配置需要从环境变量读取，并避免硬编码密钥
- **输入校验与限流不足**：建议增加参数校验、请求频率限制与审计日志
- **设备/网关鉴权与审计**：若通过 WebSocket/MQTT 接入设备端，需要验证握手阶段的 `Authorization: Bearer <token>` 与设备标识（如 `Device-Id / Client-Id`），并对 MCP `tools/call` 的参数、执行结果与失败原因做审计记录。
- **统计/趋势部分为模拟数据**：用于展示仪表盘效果；生产环境应改为真实聚合统计口径

扩展方向：
- 将数据层替换为真正的 SQLite（目前工程已安装 `sqlite` 依赖但未落用）
- 接入真实 ASR（当前语音转写主要依赖浏览器 Web Speech API）
- AI 引入更稳定的“意图结构化”方案（例如基于工具调用/函数路由）
- 如需让 ESP32 语音设备直接控制社区业务：新增 MCP 网关/适配层，把 `tools/call` 映射到本仓库现有 HTTP API（预约/报修/设备控制）。

## 7. 可扩展性总结
通过路由模块化、AI 服务层与数据层解耦的方式，平台具备后续演进能力：
- 可替换数据库实现
- 可替换 AI 提供商与模型
- 可在不破坏前端页面结构的前提下扩展更多社区服务与管理能力
